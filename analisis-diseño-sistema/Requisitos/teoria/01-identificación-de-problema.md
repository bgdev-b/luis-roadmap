# ¿Qué es el análisis de requisitos?

La ingeniería de requisitos del software es un proceso de descubrimiento,
refinamiento, modelado y especificación. Se refinan en detalle los requisitos del
sistema y el papel asignado al software.

# Tareas de análisis

El análisis de requisitos del software se puede subdividir en cinco áreas de
esfuerzo:

1. Reconocimiento del problema
2. Evaluación y síntesis
3. Modelado
4. Especificación
5. Revisión

Todos los métodos de análisis se relacionan por un conjunto de principios
operativos:

1. Debe representarse y entenderse el dominio de la información de un
problema.
2. Deben definirse las funciones que debe realizar el software.
3. Debe representarse el comportamiento del software (como
consecuencia de acontecimientos externos),
4. Deben dividirse los modelos que representan información, función y
comportamiento de manera que se descubran los detalles por capas
(o jerárquicamente).
5. El proceso de análisis debería ir desde la información esencial hasta
el detalle de la implementación.

Además de los principios operativos mencionados anteriormente, se
sugiere un conjunto de principios directrices para la ingeniería de
requerimientos:

1. Entender el problema antes de empezar a crear el modelo de
análisis.
2. Desarrollar prototipos que permitan al usuario entender como será
la interacción hombre-máquina.
3. Registrar el orden y la razón de cada requerimiento,
4. Usar múltiples planteamientos de requerimientos.
5. Priorizar los requerimientos.
6. Trabajar para eliminar la ambigüedad.

Un ingeniero de software que se apegue a estos principios es muy probable
que desarrolle una especificación de software que represente un excelente
fundamento para el diseño.

# Funciones y habilidades del analista

La función principal de un analista del software (o ingeniero de requisitos
es llevar a cabo las actividades necesarias para cumplir con las cinco áreas de
esfuerzo descritas en la sección anterior. Para lo cual hace uso de las siguientes
técnicas:

1. Entrevistas
2. Talleres
3. Observación
4. Encuestas
5. Revisión documental
6. Uso de especificaciones formales para requerimientos (formatos
estándar de documentos, UML, etc.)

# Identificación del Problema

## Definición

La identificación del problema consiste en comprender una situación no deseada, determinar sus causas y evaluar su impacto sobre los usuarios u organizaciones afectadas.

El objetivo es descubrir el problema real antes de diseñar una solución.

---

## Elementos Clave

### Situación Actual

¿Cómo funciona el proceso actualmente?

### Problema

¿Qué está fallando?

### Impacto

¿Qué consecuencias genera?

### Causa Raíz

¿Por qué ocurre?

### Afectados

¿Quiénes sufren el problema?

---

## Explicación Feynman

Identificar un problema es parecido a visitar un médico.

El médico no receta medicamentos inmediatamente.

Primero investiga:

* Qué síntomas existen.
* Cuál es la causa.
* Qué consecuencias tiene.

Solo entonces propone un tratamiento.

En los sistemas ocurre exactamente lo mismo.

---

## Ejemplo: Gestor de Turnos

### Situación Actual

Los clientes reservan citas por WhatsApp.

### Problema

Se producen conflictos de horarios.

### Impacto

Pérdida de tiempo y mala experiencia para los clientes.

### Causa Raíz

No existe un mecanismo automático de validación.

### Afectados

* Clientes.
* Empleados.
* Negocio.
